# 事务的四大特性 (ACID)

这是数据库领域的“金科玉律”，面试和考试必考：

- **原子性 (Atomicity)**：事务是最小的单位，不可再分。要么全做，要么全不做。
    
- **一致性 (Consistency)**：事务执行前后，数据库从一个一致性状态转换到另一个一致性状态（比如转账前后，两人总额不变）。
    
- **隔离性 (Isolation)**：多个事务并发执行时，彼此隔离，互不干扰（防止你转账时，别人也在给你存钱导致数据乱套）。
    
- **持久性 (Durability)**：一旦事务提交，它对数据库的改变就是永久性的，即使系统崩溃也不会丢失。


# 基础语法与流程

在 MySQL 中，手动控制事务的流程非常固定：

## 基础语法：


```
-- 1. 开启事务（告诉数据库：接下来的操作先别急着入库）
START TRANSACTION; 
-- 或者使用 BEGIN;

-- 2. 执行具体的业务逻辑
UPDATE account SET money = money - 100 WHERE name = '张三';
UPDATE account SET money = money + 100 WHERE name = '李四';

-- 3. 提交事务（确认没问题，正式写入磁盘）
COMMIT;

-- 或者：如果发现出错了（比如钱不够减）
-- ROLLBACK; -- 回滚事务，撤销上面所有的修改
```


## 事务在项目中的实际应用

在学习的 **“苍穹外卖”** 或 **“黑马点评”** 项目中，事务无处不在。

- **下单业务**：
    
    1. 插入订单表（Order）。
        
    2. 插入订单详情表（OrderDetail）。
        
    3. 扣减菜品库存（Stock）。
        
    
    - 这三步必须封装在一个事务里。如果库存扣减失败（卖完了），那么之前的订单插入也必须撤销。

## ROLLBACK 回滚

`ROLLBACK` 是用于撤销当前事务中所有未提交修改的操作。它将数据库的状态恢复到事务开始（执行 `START TRANSACTION`）那一刻，或者恢复到你手动设置的某个“存档点（Savepoint）”。

它是实现事务 **原子性（Atomicity）** 的核心手段：保证一组操作要么全成，要么由于某个环节出错，全部像没发生过一样。

### 基础语法结构

在 MySQL 等关系型数据库中，回滚通常配合事务控制语句使用：

```
-- 1. 开启事务
START TRANSACTION;

-- 2. 执行一系列 SQL 操作
-- 操作 1 ...
-- 操作 2 ...

-- 3. 撤销操作（回滚）
ROLLBACK; 

-- 4. 或者回滚到特定存档点
-- ROLLBACK TO SAVEPOINT 存档点名称;
```

### 回滚是怎么实现的？

数据库并不是真的“删除”了你刚才的操作，而是利用了 **Undo Log（回滚日志）**：

- **逻辑记录**：当你执行 `INSERT` 时，Undo Log 会记录一条对应的 `DELETE`；当你执行 `UPDATE` 时，它会记录这一行修改前的“老样子”。
    
- **指令逆转**：当你下达 `ROLLBACK` 命令时，数据库会从 Undo Log 里读出这些逆向操作并执行一遍，从而把数据“推回”原来的样子。
    
- **自动回滚**：如果你的数据库连接突然断开（比如 Java 程序崩溃），数据库引擎检测到事务没提交，也会自动执行 `ROLLBACK` 来保护数据。
