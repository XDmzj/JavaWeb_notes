# 题解

```sql
select
round(
    sum(case when q2.device_id is not null then 1 else 0 end)
    /count(*)
    ,4)
 as avg_ret
from (SELECT DISTINCT device_id, date FROM question_practice_detail) q1
left join (SELECT DISTINCT device_id, date FROM question_practice_detail) q2
on q1.device_id = q2.device_id
and q2.date = date_add(q1.date,interval 1 day)
;		
```

得到的表中次日留存的用户的q2.device_id非空

## sum的逻辑
确定数据池

`SUM` 并不是对整张表求和，而是对 **经过过滤（WHERE）和分组（GROUP BY）后** 剩下的那一列数值进行操作。

- 如果你有 `GROUP BY school`，那么 `SUM` 会为每个学校开启一个独立的“记账本”。

---

自动忽略 NULL

这是 `SUM` 最核心的逻辑。在 SQL 中，`NULL` 代表“未知”，而 `1 + 未知 = 未知`。为了不让整个求和结果变成 `NULL`，`SUM` 会在计算时**自动跳过**所有 `NULL` 值。

- **例子**：一组数据为 `[10, 20, NULL, 30]`。
    
- **计算逻辑**：$10 + 20 + 30 = 60$。
    
- **注意**：如果整列全是 `NULL`，`SUM` 的结果才会是 `NULL`。
---

⭐逐行累加

`SUM` 会遍历数据池里的每一行，取出数值并加到累加器上。

在这个特殊场景中：

`SUM(CASE WHEN p2.device_id IS NOT NULL THEN 1 ELSE 0 END)`

1. **转换层**：每一行先经过 `CASE` 处理。要么变成 `1`，要么变成 `0`。
2. **数据流**：`SUM` 此时看到的不再是 `device_id`，而是一串 `[1, 1, 0, 1, 0...]`。
3. **累加层**：
    - 碰到 `1`：总数 +1（代表抓到一个留存用户）。
    - 碰到 `0`：总数 +0（代表这个坑位没对上人，不加分）。
4. **结果层**：输出最终的总分。


## select distinct device_id ,date from question_practice_detail的原因

在 `question_practice_detail` 这张表里，记录的是用户的**每一次答题流水**。如果一个用户在同一天内做了 5 道题，这张表就会产生 5 行数据。

使用 `SELECT DISTINCT device_id, date` 的目的只有一个：**将“答题流水表”降级为“用户活跃表（日活表）”**。


**如果不加 DISTINCT：**

- **q1 (4月8日)** 有 2 行数据。
    
- **q2 (4月9日)** 有 3 行数据。
    
- **JOIN 结果**：由于 SQL 的连接机制，`q1` 的每一行都会去匹配 `q2` 的每一行。$2 \times 3 = 6$。
    
- **后果**：原本只是“一个人留存了”，结果在 `SUM` 时却加了 **6 次 1**。这显然会让分子变得大得离谱。


## date.add()  \  interval 1 day

核心语法
`DATE_ADD(start_date, INTERVAL expr unit)`

- **`p1.date`**：起始日期（通常是表 `p1` 中的某一列）。
    
- **`INTERVAL 1 DAY`**：时间增量。它告诉数据库：“请往后移动 1 天”

**执行结果：** 如果 `p1.date` 是 `2026-04-08`，执行后的结果就是 `2026-04-09`。

还有：
**运算符法**：`p1.date + INTERVAL 1 DAY` （更简洁）。
**专门减法**：如果你想往前推一天，可以用 `DATE_SUB(p1.date, INTERVAL 1 DAY)`
