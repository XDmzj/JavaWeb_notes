# 题目：
现在运营想要分别查看学校为山东大学或者性别为男性的用户的device_id、gender、age和gpa数据，请取出相应结果，结果不去重。

# 题解
~~~sql
select 
    device_id, gender, age, gpa
from user_profile
where university='山东大学'

union all

select 
    device_id, gender, age, gpa
from user_profile
where gender='male'
~~~


## 知识点


### union是什么

简单来说，`UNION` 是 SQL 中用来**纵向合并**结果集的“胶水”。

如果说 `JOIN` 是把两张表**横向左右拼接**（增加列），那么 `UNION` 就是把两个查询结果**纵向上下堆叠**（增加行）。

---

#### 1. 核心定义

`UNION` 操作符用于合并两个或多个 `SELECT` 语句的结果集。

- **UNION**：合并结果并**自动去重**。
    
- **UNION ALL**：合并结果并**保留所有行**（包括重复行）。

---

#### 2. 使用的“硬性门槛”

并不是任意两个查询都能 `UNION`。要成功合并，必须满足以下三个条件：

1. **列数相同**：第一个 `SELECT` 选了 4 列，第二个也必须选 4 列。
    
2. **数据类型相似**：对应位置的列，数据类型必须兼容（比如都是数字或都是字符）。
    
3. **顺序一致**：列的语义顺序应该一致，否则合并出来的表格数据会张冠李戴。

#### 3. UNION vs UNION ALL：谁更常用？

| **特性**   | **UNION**             | **UNION ALL**    |
| -------- | --------------------- | ---------------- |
| **重复处理** | **去重**（类似 `DISTINCT`） | **不去重**（全盘接收）    |
| **性能**   | **较慢**（需要额外排序比对来去重）   | **快**（直接堆叠，效率极高） |
| **结果数量** | 可能比两个子查询总和少           | 等于两个子查询总和        |




### or和union

大多数情况下，用 `OR` 会更简洁、更符合人类的直觉。
但是，`OR` 和 `UNION ALL` 在 SQL 中并不完全等价。选择哪一个，取决于你对**“重复数据”**的处理态度以及对**“查询性能”**的要求。
以下是它们的核心区别：

---

#### 1. 结果集：是否包含“重复项”？

这是两者最直观的区别：

- **使用 `OR`**： 如果一个用户既是**山东大学**又是**男性**，他只会在结果中出现 **1 次**。
    
    > 逻辑：只要满足其中一个条件，这一行就被选中。
    
- **使用 `UNION ALL`**： 如果一个用户既是**山东大学**又是**男性**，他会在结果中出现 **2 次**（一次来自第一个 SELECT，一次来自第二个）。
    
    > 逻辑：它是两个结果集的简单堆砌。

**结论：** 如果你想要的是一份**去重后的名单**，用 `OR` 更直接；如果你需要统计**总的人次触发频率**，`UNION ALL` 才有意义。

---

#### 2. 性能：谁跑得更快？

在处理海量数据（千万级以上）时，性能差异会显现出来：

- **`OR` 的痛点**： 数据库的优化器有时很难同时利用两个不同列的索引。比如你给 `university` 编了号，也给 `gender` 编了号，但在执行 `WHERE university='...' OR gender='...'` 时，数据库可能会放弃使用索引，转而进行**全表扫描**，导致查询变慢。
    
- **`UNION ALL` 的优势**： 它把任务拆成了两个独立的查询。第一个查询可以用 `university` 的索引，第二个可以用 `gender` 的索引，最后把结果合并。在数据量极大且索引丰富的情况下，这种方式往往比 `OR` 快得多。
