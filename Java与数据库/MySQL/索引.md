# 索引的核心作用

- **大幅提升查询速度（SELECT）**：这是最直观的作用，通过 B+ 树等数据结构，将查询时间复杂度从 $O(N)$ 降至 $O(\log N)$。
    
- **加速排序与分组**：当 `ORDER BY` 或 `GROUP BY` 涉及索引列时，数据库可以直接利用索引的有序性，无需额外的文件排序。
    
- **唯一性约束**：通过唯一索引，可以强制保证列中数据的唯一性。


# 索引的数据结构是 B+ 树 

大多数关系型数据库（如 MySQL 的 InnoDB）使用 **B+ 树** 作为索引的默认数据结构：

- **分层存储**：数据存储在叶子节点，非叶子节点只存储键值（导航信息），这使得树的深度极小，减少了磁盘 I/O 次数。
    
- **范围查询友好**：叶子节点之间通过双向链表连接，在进行范围查找（如 `age > 20 AND age < 30`）时效率极高。

# 索引的性能优化

索引虽好，但并非越多越好。

- **维护成本高**：每次对表进行 `INSERT`、`UPDATE`、`DELETE` 操作时，数据库都必须同时更新相关的索引，这会降低写入性能。
    
- **占用存储空间**：索引本身也是存储在磁盘上的文件。

# 语法
## 创建索引 (CREATE)

可以在建表时指定，也可以后期添加。

- **普通索引**：最基本的索引。

```
CREATE INDEX idx_user_name ON users(name);
-- 或者
ALTER TABLE users ADD INDEX idx_user_name(name);    
```   

- **唯一索引 (UNIQUE)**：保证列值唯一，允许为 NULL。

```
CREATE UNIQUE INDEX idx_user_phone ON users(phone);
```

- **联合索引 (Composite)**：多个字段组合。

```
CREATE INDEX idx_name_age ON users(name, age);
```

- **主键索引**：通常在建表时通过 `PRIMARY KEY` 自动创建。

## 查看索引 (SHOW)

查看表中已有的索引及其详细信息（如序列号、是否唯一等）。

```
SHOW INDEX FROM users;
```

## 删除索引 (DROP)

```
DROP INDEX idx_user_name ON users;
-- 或者
ALTER TABLE users DROP INDEX idx_user_name;
```



# 索引的生效

索引本质上是改变了数据库查询的方法，将全表遍历改为B+树查找，效率提升（大量数据下）

你调用select的时候就自动使用索引了，不用专门调用索引
